## §2 バージョン管理ルール

### §2.1 メタデータブロック

すべての管理対象ドキュメントは、ファイル冒頭に以下のメタデータブロックを**必須**で記載します:

#### 標準メタデータブロック形式

```markdown
---
ファイル名: <filename>
バージョン: Ver.X.Y-Z
作成日: YYYY-MM-DD
最終更新日: YYYY-MM-DD
ステータス: <記号> <ステータス名>
分類: <内部専用 | 機密 | 公開>
---
```

#### フィールド説明

| フィールド | 必須 | 説明 | 例 |
|-----------|------|------|-----|
| `ファイル名` | ✅ | ドキュメントのファイル名 | `docs/SPECIFICATION.md` |
| `バージョン` | ✅ | ドキュメントのバージョン | `Ver.1.2-5` |
| `作成日` | ✅ | 初版作成日 | `2026-03-10` |
| `最終更新日` | ✅ | 最後に変更した日付 | `2026-03-15` |
| `ステータス` | ✅ | ドキュメント自体の状態（§2.5参照） | `✅ 確定` |
| `分類` | ✅ | 公開レベル | `内部専用` / `機密` / `公開` |

**重要:**
- `ステータス` フィールドには**ドキュメント自体の状態**のみを記載（詳細は §2.5 参照）
- 日付フォーマットは `YYYY-MM-DD` に統一する

---

### §2.2 変更履歴の管理方法

#### 🚫 各ドキュメントへの変更履歴テーブル記載は禁止

すべてのドキュメントの変更履歴は、**`docs/DOCUMENT_CHANGELOG.md`** で一元管理します。

#### ✅ 各ドキュメント末尾には参照リンクを記載

各ドキュメントの末尾に、以下の参照セクションを記載します:

```markdown
## 📝 変更履歴

本ドキュメントの変更履歴は、以下のファイルで一元管理されています:

👉 **[docs/DOCUMENT_CHANGELOG.md](./DOCUMENT_CHANGELOG.md)** を参照してください。
```

---

### §2.3 バージョン番号の扱い

バージョン番号は `VERSIONING.md` の累積ルールに準拠します:

- **メジャー番号 (X)**: プラットフォームバージョンと連動
- **マイナー番号 (Y)**: ドキュメントの構造変更や大規模改訂
- **ビルド番号 (Z)**: ドキュメントごとに独立管理、**リセット禁止**

---

### §2.4 変更履歴の記載ルール

`docs/DOCUMENT_CHANGELOG.md` に変更履歴を記載する際は、以下のルールに従います:

1. **新しい変更履歴はテーブルの最上部に追加**
2. **日付は `YYYY-MM-DD` 形式で統一**
3. **変更内容は簡潔に1行で記述** (複数項目を列挙可)
4. **変更者は GitHub アカウント名を記載**
5. **バージョン番号は各ドキュメントのメタデータブロックと一致させる**

### §2.5 ステータス管理ルール

#### 基本方針

**ドキュメントのステータスはプラットフォームバージョンと連動しません。**

- ドキュメントは開発の**リファレンス（参照基準）**であり、実装より先行する
- ドキュメントに記載された仕様に基づいて開発が進行する
- ドキュメント自体の完成度・承認状態を独立して管理する

#### ステータス値の定義

すべての管理対象ドキュメントのメタデータブロックには、以下の標準ステータスのいずれかを記載します:

| ステータス | 記号 | 意味 | 使用条件 |
|-----------|------|------|----------|
| **策定中** | 🔧 | ドキュメントを執筆中、内容が未確定 | 初版作成時、大規模改訂中 |
| **暫定運用** | ⚠️ | 仮運用中、変更の可能性あり | 試験運用期間中 |
| **確定** | ✅ | 正式決定済み、原則変更しない | 正式承認後、安定運用中 |
| **廃止予定** | ⚠️ | 将来的に削除予定 | 後継ドキュメントが存在する場合 |
| **廃止** | ❌ | 使用禁止、参照のみ | アーカイブ目的で保持 |

**注意事項:**
- プラットフォームバージョン（Ver.1.4-pre など）はステータスフィールドに記載しない
- プラットフォームバージョンはドキュメント本文内で言及する

#### ステータスの記載形式

メタデータブロックでは、以下の形式でステータスを記載します:

```markdown
ステータス: 🔧 策定中
ステータス: ⚠️ 暫定運用
ステータス: ✅ 確定
ステータス: ✅ 確定（第2版）
ステータス: ⚠️ 廃止予定（2026-06-30 終了予定）
```

**ルール:**
- **記号 + ステータス名**の形式を使用
- 必要に応じて括弧内に補足情報を記載（期限、版数など）
- 簡潔に1行で記述

#### ステータス遷移フロー

```
[策定中 🔧] ─────→ [確定 ✅] ─→ [廃止予定 ⚠️] → [廃止 ❌]
                     ↑            ↓
[暫定運用 ⚠️] ───────┘       (改訂時: 確定 → 策定中)
```

**遷移パターン:**

1. **策定中から確定へ**
   ```
   策定中 → 確定
   ```

2. **暫定運用から確定へ**
   ```
   暫定運用 → 確定
   ```

3. **改訂パターン（確定から策定中へ戻る）**
   ```
   確定 → 策定中（改訂開始） → 確定
   ```

4. **廃止パターン**
   ```
   確定 → 廃止予定 → 廃止
   ```

**重要なポイント:**
- 「策定中」と「暫定運用」は**独立した並行パス**であり、互いに遷移しない
- 「確定」ステータスでも、必要に応じて「策定中」に戻して改訂可能
- 軽微な修正（誤字修正、リンク更新など）は「確定」のまま実施可能

#### ステータス変更のルール

1. **ステータス変更時は必ずバージョン番号をインクリメント**
   - マイナー番号 (Y): 大規模改訂、ステータス変更を伴う場合
   - ビルド番号 (Z): 軽微な変更、ステータス変更なし
   
2. **ステータス変更は `DOCUMENT_CHANGELOG.md` に記録**
   ```markdown
   | Ver.1.1-1 | 2026-03-11 | ステータスを「策定中」→「確定」に変更 | fqwink |
   | Ver.1.2-1 | 2026-03-15 | ステータスを「暫定運用」→「確定」に変更 | fqwink |
   ```

3. **確定ステータスへの変更には承認が必要**
   - Adlaire Group の承認者による承認が必要
   - 承認記録を `DOCUMENT_CHANGELOG.md` に記載
   ```markdown
   | Ver.1.2-1 | 2026-03-15 | ステータスを「確定」に変更（承認者: @reviewer）| fqwink |
   ```

4. **廃止予定ステータスには終了予定日を明記**
   ```markdown
   ステータス: ⚠️ 廃止予定（2026-06-30 終了予定、後継: SPECIFICATION_v2.md）
   ```

## §3 禁止事項

以下の操作は**永続的に禁止**されます:

1. **`CHANGELOG.md` の作成・削除** (プラットフォーム本体の変更履歴は別管理)
2. **`docs/DOCUMENT_CHANGELOG.md` の削除** (ドキュメント変更履歴の一元管理元)
3. **保護対象ドキュメント (`🚫削除` マーク) の削除**
4. **各ドキュメントへの変更履歴テーブルの記載** (一元管理に統一)
5. **誤って作成された禁止ファイルは即座に削除**

## 📝 ドキュメント更新時の手順

### ドキュメント編集時のフロー

1. **対象ドキュメントを編集**
2. **メタデータブロックのバージョン番号と最終更新日を更新**
   ```markdown
   バージョン: Ver.0.5-2  ← ビルド番号をインクリメント
   最終更新日: 2026-03-11  ← 今日の日付に更新
   ```
3. **`docs/DOCUMENT_CHANGELOG.md` に変更履歴を追加**
   - 該当ドキュメントのテーブルの**最上部**に新しい行を追加
   ```markdown
   | Ver.0.5-2 | 2026-03-11 | セキュリティ方針の詳細を追加、図を更新 | fqwink |
   ```
4. **コミット & プッシュ**
