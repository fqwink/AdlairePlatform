# AdlairePlatform フレームワークルールブック

## §1 目的

本ルールブックは、AdlairePlatform におけるフレームワークの設計・開発・運用に関する統一規則を定める。

---

## §2 基本方針

### §2.1 エンジン駆動モデルフレームワーク

| 方針 | 内容 |
|------|------|
| 設計モデル | すべてのフレームワークはエンジン駆動モデルを採用する |
| 1 エンジン 5 ファイル原則 | 1 フレームワーク = 1 エンジン。5 つのモジュラーコンポーネント（Core / Api / Utils / Interface / Class）で構成 |
| コンポーネント内設計 | 1 コンポーネント（ファイル）に複数クラスを格納 |
| フレームワーク間依存 | 依存ゼロ。フレームワーク間の直接 import / 参照を禁止 |
| 外部依存 | 外部ライブラリへの依存ゼロ |

### §2.2 設計原則

| 原則 | 内容 |
|------|------|
| エンジン単位の自己完結 | 5 ファイル一体で自己完結する。他エンジン（フレームワーク）の存在を前提としない |
| フレームワーク間依存ゼロ | フレームワーク間の直接 import / 参照を禁止。唯一の例外として ACS.Interface が定義する通信契約型への参照を許可する（§4.6 参照） |
| 外部依存ゼロ | 外部ライブラリ不要で動作する。言語標準ライブラリのみ使用可 |
| インターフェース契約 | フレームワーク間の通信は Interface ファイルが定義する契約に基づく。具象クラスへの直接依存を禁止 |
| API 経由 | フロントエンド ↔ バックエンド間の通信はすべて API 経由とする |
| 責務分離 | バックエンドはデータ提供に専念、描画はフロントエンド側で実行する |

### §2.3 言語要件

| 言語 | バージョン | 用途 |
|------|----------|------|
| PHP | 8.3 以上 | サーバ（ASS） |
| TypeScript | Deno 2.x 以上 | バックエンド・クライアント |
| JavaScript | ES6+ | フロントエンド（AEB） |
| CSS | CSS3 | フロントエンド（ADS） |

## §3 アーキテクチャ分類

### §3.1 分類定義

```
┌──────────────────────────────────────────────────────┐
│                  AdlairePlatform                      │
│                                                      │
│  ┌────────────┐  ┌──────────┐  ┌──────────────────┐  │
│  │フロントエンド│  │クライアント│  │   バックエンド    │  │
│  │ AEB, ADS   │  │   ACS    │  │ APF,ACE,AIS,ASG,AP│  │
│  └────────────┘  └─────┬────┘  └──────────────────┘  │
│                        │                              │
│                        ▼                              │
│                  ┌──────────┐                          │
│                  │  サーバ   │                          │
│                  │   ASS    │                          │
│                  └──────────┘                          │
└──────────────────────────────────────────────────────┘
```

### §3.2 分類一覧

| 分類 | 役割 | フレームワーク |
|------|------|--------------|
| フロントエンド | UI 描画・ブラウザ側処理 | AEB, ADS |
| クライアント | サーバとの通信抽象化 | ACS |
| バックエンド | ビジネスロジック・CMS 機能 | APF, ACE, AIS, ASG, AP |
| サーバ | 認証・データストレージ・ファイル管理 | ASS |

### §3.3 分類規則

| 規則 | 内容 |
|------|------|
| 通信経路 | バックエンド ↔ サーバの通信は必ず ACS 経由 |
| 直接参照禁止 | フロントエンドからサーバのクラス・関数を直接呼び出すことは禁止 |
| ACS の役割 | サーバ実装の詳細を隠蔽し、統一インターフェースを提供する |
| フロントエンドの独立性 | サーバ言語に依存しない |
| バックエンドの独立性 | ブラウザ API に依存しない（Deno 環境分岐は許容） |
| サーバの独立性 | 他フレームワークに依存しない。ACS のインターフェース契約のみに準拠 |

## §4 エンジン駆動モデル

### §4.1 モデル特徴

エンジン駆動モデルは、AdlairePlatform 独自のフレームワーク設計パターンである。

| 特徴 | 説明 |
|------|------|
| **1 エンジン 5 ファイル原則** | 1 フレームワーク = 1 エンジン。5 つのモジュラーコンポーネント（Core / Api / Utils / Interface / Class）が一体となり、単一のエンジンを構成する |
| **モジュラーコンポーネント原則** | 各ファイルはエンジンの交換可能な構成部品（モジュラーコンポーネント）である。1 コンポーネント内に複数クラスの配置を許容する。各コンポーネントには主たる役割があるが、エンジン全体の設計意図に沿う範囲でコンポーネント間の役割配分は柔軟に調整できる |
| **エンジン内コンポーネント間協調** | 同一エンジン内の 5 コンポーネントは相互参照が可能。他エンジンへの参照は禁止 |
| **外部依存ゼロ** | サードパーティライブラリ・外部パッケージへの依存を禁止する。言語標準ライブラリのみ使用可 |
| **エンジン単位の自己完結** | 5 ファイル一体で自己完結する。他エンジン（フレームワーク）の存在を前提としない |
| **フレームワーク間依存ゼロ** | エンジン間の直接 import / 参照を禁止。唯一の例外として ACS.Interface が定義する通信契約型への参照を許可する（§4.6 参照） |
| **統一命名体系** | ファイル名 `{略称}.{エンジン名}.{拡張子}` により、所属と役割が即座に判別可能 |

### §4.2 コンポーネント役割ガイドライン

各モジュラーコンポーネントには主たる役割がある。エンジン全体の設計意図に沿う範囲で、役割配分は柔軟に調整できる。

| コンポーネント | 主たる役割 | 配置の目安 |
|--------------|----------|----------|
| **Core** | ビジネスロジック | サービスクラス、ドメインロジック、状態管理 |
| **Api** | 外部との入出力境界層 | エンジンと外部世界の境界処理（§4.2.1 参照） |
| **Utils** | 補助機能 | ユーティリティ関数、ヘルパークラス、共通処理 |
| **Interface** | 契約定義 | インターフェース、抽象クラス、型契約 |
| **Class** | データモデル | データクラス、値オブジェクト、列挙型、型定義 |

#### §4.2.1 Api コンポーネントの分類別責務

Api はエンジンと外部世界の入出力境界を担う。対象となる「外部」は分類により異なる。

| 分類 | Api の外部対象 | 具体的な責務 |
|------|-------------|------------|
| サーバ（ASS） | HTTP クライアント | エンドポイント定義、ルーティング、リクエスト解析、レスポンス生成 |
| クライアント（ACS） | サーバ API | HTTP クライアント処理、リクエスト構築、レスポンス変換 |
| バックエンド（APF 等） | ACS 経由のサーバ | ACS を介したサーバアクセスのアダプター処理 |
| フロントエンド（AEB） | ブラウザ DOM | DOM イベントバインディング、ユーザー操作の受付・委譲 |
| フロントエンド（ADS） | CSS カスタムプロパティ | テーマ変数公開、外部からのスタイル制御インターフェース |

### §4.3 モジュラーコンポーネント構成

1 エンジン = **5 モジュラーコンポーネント**で構成する。各コンポーネントはエンジンの交換可能な構成部品であり、単独では機能しない。5 コンポーネントが一体となることでエンジンとして動作する。

### §4.4 例外規定

| 条件 | 許容内容 | 承認 |
|------|---------|------|
| 構成の縮小（1〜4 ファイル） | 単一責務のフレームワークに限り許容 | Adlaire Group 承認必須 |
| 構成の拡張（6〜7 ファイル） | 明確な責務分離理由がある場合に許容 | Adlaire Group 承認必須 |
| 8 ファイル以上 | **禁止** — 新規フレームワーク化を検討する | ❌ 禁止 |

### §4.5 フレームワーク間依存規則

フレームワーク間の直接依存はゼロとする。

| 規則 | 内容 |
|------|------|
| フレームワーク間依存 | 禁止。フレームワーク間で直接 import / 参照してはならない |
| 唯一の例外 | ACS.Interface が定義する通信契約型への参照のみ許可する（§4.6 参照） |
| 相互依存 | 禁止 |

### §4.6 共有層定義

フレームワーク間で共有が必要な型定義・インターフェースは、ACS（Adlaire Client Services）の Interface コンポーネントに集約する。

| 規則 | 内容 |
|------|------|
| 集約先 | ACS.Interface.ts（TypeScript）/ ACS.Interface.php（PHP） |
| 集約対象 | 通信契約型（リクエスト/レスポンス型、API エンドポイント契約、共通エラー型） |
| 参照許可範囲 | ACS.Interface のみ。ACS の他コンポーネント（Core / Api / Utils / Class）への参照は禁止 |
| 参照方向 | 他エンジン → ACS.Interface の一方向のみ。ACS.Interface が他エンジンを参照してはならない |
| 集約禁止 | ビジネスロジック固有の型、特定エンジン内部でのみ使用する型は各エンジンの Interface / Class に定義する |

```
  APF ──→ ACS.Interface ←── ACE
  AIS ──→ ACS.Interface ←── ASG
   AP ──→ ACS.Interface ←── AEB
                ↑
               ASS（PHP 側は ACS.Interface.php の契約に準拠）
```

## §5 命名規則

### §5.1 フレームワーク略称

3 文字固定。正式名称の頭文字に基づく英大文字。全フレームワーク間で一意。`AP` のみ 2 文字例外。

### §5.2 ファイル

形式: `{略称}.{エンジン名}.{拡張子}`（PascalCase）

拡張子: `.php` / `.ts` / `.js` / `.css`

### §5.3 名前空間・エクスポート

PHP: `{略称}\{エンジン名}\{クラス名}`

TypeScript: named export のみ。デフォルトエクスポート禁止。

### §5.4 ディレクトリ

`Framework/{略称}/`（略称は大文字）

## §7 API 規則

すべてのサーバ通信は ACS の `AdlaireClient` インターフェース経由とする。サーバ側 API は ASS が一元管理する。

| 規則 | 内容 |
|------|------|
| 通信プロトコル | HTTP/HTTPS |
| データ形式 | JSON |
| 認証 | すべての API リクエストに認証を要求（公開 API を除く） |

禁止事項:

- ACS を経由しないサーバアクセス
- フロントエンドからサーバ言語の直接呼び出し
- バックエンドからブラウザ DOM の直接操作
- ASS 以外の独自サーバ API エンドポイント作成

## §8 インポート規則

### §8.1 PHP

名前空間プレフィックス → 物理ファイルの直接マッピング。PSR-4 非準拠（1 ファイル = 複数クラスのため）。

### §8.2 TypeScript

| 規則 | 内容 |
|------|------|
| 形式 | named import のみ |
| デフォルトエクスポート | 禁止 |
| パス | 相対参照 |
| 拡張子 | `.ts` を必ず含める |
| re-export | 重複クラス統合時に使用。統合元が正 |
| 共有型の参照 | `ACS.Interface.ts` からの import のみ、フレームワーク間で許可（§4.6 参照） |
