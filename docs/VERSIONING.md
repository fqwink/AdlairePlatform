## バージョニング方式

**採用方式**: セマンティックバージョニングベースの累積バージョニング

Adlaire Group では、セマンティックバージョニング（Semantic Versioning）をベースとした**累積バージョニング**を採用する。

| 方針 | ステータス |
|------|-----------|
| セマンティックバージョニングをベースとする | ✅ 確定 |
| ビルド/リビジョンを総累積で管理する | ✅ 確定 |
| ビルド/リビジョンをリセットしてはならない | ❌ **禁止**（理由は下記「リセット禁止規則」を参照） |
| `Ver.` 接頭辞により明確に表記する | ✅ 確定 |

---

## 特徴

### セマンティックバージョニングの原則を継承

- メジャー、マイナー、ビルド/リビジョンの 3 階層で構成する
- 後方互換性を明確に区別する

### 累積バージョニングの追加

- リビジョン番号は累積的に増加させる
- 過去からの総変更回数を表現する

### 明確な表記形式

- `Ver.{メジャー}.{マイナー}-{ビルド/リビジョン}`
- 例: `Ver.1.5-7`

---

## バージョン番号の構成

### 基本構造

```
Ver.{メジャー}.{マイナー}-{ビルド/リビジョン}
```

**例**:
```
Ver.1.5-7
 │  │ │ │
 │  │ │ └─ ビルド/リビジョン番号（累積・リセット禁止）
 │  │ └─── マイナーバージョン番号
 │  └───── メジャーバージョン番号
 └──────── バージョン接頭辞（固定）
```

---

## 各要素の定義

### 1. Ver.（バージョン接頭辞）

| 項目 | 内容 | ステータス |
|------|------|-----------| 
| 表記 | `Ver.`（固定） | ✅ 確定 |
| 役割 | バージョン番号であることを明示する | ✅ 確定 |
| 必須 | すべてのバージョン表記において必須とする | ✅ 確定 |

### 2. メジャーバージョン（Major Version）

| 項目 | 内容 | ステータス |
|------|------|-----------| 
| 初期値 | `0` または `1` | ✅ 確定 |
| 増加条件 | 後方互換性のない変更が行われた場合に増加する | ✅ 確定 |
| 範囲 | `0` ～ 無制限 | ✅ 確定 |
| リセット | メジャー変更時も、マイナーとリビジョンをリセットしてはならない（累積） | ❌ **禁止** |

### 3. マイナーバージョン（Minor Version）

| 項目 | 内容 | ステータス |
|------|------|-----------| 
| 初期値 | `0` | ✅ 確定 |
| 増加条件 | 後方互換性を保った変更（機能追加、改良、中〜高重要度のバグ修正、緊急バグ修正）が行われた場合に増加する | ✅ 確定 |
| 範囲 | `0` ～ 無制限 | ✅ 確定 |
| リセット | マイナー変更時も、リビジョンをリセットしてはならない（累積） | ❌ **禁止** |

### 4. ビルド/リビジョン（Build/Revision）

| 項目 | 内容 | ステータス |
|------|------|-----------| 
| 初期値 | `1` | ✅ 確定 |
| 増加条件 | 微調整、軽微なバグ修正、ドキュメント更新、コメント修正、または機能変更を伴わないリファクタリングが行われた場合に増加する | ✅ 確定 |
| 範囲 | `1` ～ 無制限 | ✅ 確定 |
| 特性 | **総累積管理** ─ プロジェクト開始からの総変更回数を示す | ✅ 確定 |
| リセット | **いかなる場合もリセットしてはならない** | ❌ **禁止**（詳細は下記参照） |

---

## ⚠️ ビルド/リビジョン：リセット禁止規則（総累積管理）

> ❌ **これは Adlaire Group バージョニングの根幹ルールである。いかなる場合も例外はない。**

| 禁止行為 | 説明 | ステータス |
|---------|------|-----------|
| メジャーアップ時のリビジョンリセット | `Ver.1.5-20 → Ver.2.0-1` のようなリセットは禁止する | ❌ 禁止 |
| マイナーアップ時のリビジョンリセット | `Ver.1.4-15 → Ver.1.5-1` のようなリセットは禁止する | ❌ 禁止 |
| 任意タイミングでのリビジョンリセット | 運用上の理由によるリセットもすべて禁止する | ❌ 禁止 |
| リビジョン番号の付け直しまたは振り直し | 過去に遡っての番号変更も禁止する | ❌ 禁止 |

### 正しい運用（総累積管理）

```
✅ 正しい例（リセットなし・総累積）
Ver.0.1-1  → Ver.0.1-2 → Ver.0.2-3 → Ver.1.0-4 → Ver.1.1-5 → Ver.2.0-6

❌ 誤った例（リセットあり）
Ver.0.1-1  → Ver.0.1-2 → Ver.0.2-1 → Ver.1.0-1 → Ver.1.1-1 → Ver.2.0-1
                                ↑リセット禁止      ↑リセット禁止
```

### 総累積管理の意味

ビルド/リビジョン番号は **「プロジェクト開始からの総変更回数」** を示す。

- リビジョン番号を見れば、そのバージョンがプロジェクト全体で何回目の変更であるかを即座に把握できる
- 番号の連続性がプロジェクト全体の変更履歴における完全性を保証する
- リセットを行うと履歴の追跡可能性が失われるため、**絶対に行ってはならない**

## バージョンアップ基準

### メジャーバージョンアップ基準

| 項目 | 内容 | ステータス |
|------|------|-----------| 
| 変更条件 | 後方互換性のない破壊的変更が行われた場合に適用する | ✅ 確定 |
| リビジョンの扱い | 累積的に増加させる（リセットは禁止） | ❌ リセット禁止 |

---

### マイナーバージョンアップ基準

| 項目 | 内容 | ステータス |
|------|------|-----------|
| 変更条件 | 後方互換性を保った機能追加、機能改良、バグ修正（中〜高）、または緊急バグ修正が行われた場合に適用する | ✅ 確定 |
| リビジョンの扱い | 累積的に増加させる（リセットは禁止） | ❌ リセット禁止 |

---

### ビルド/リビジョンアップ基準

| 増加条件 | ステータス |
|---------|-----------|
| 内部的な微調整 | ✅ 確定 |
| 緊急バグ修正（軽微） | ✅ 確定 |
| バグ修正（軽微） | ✅ 確定 |
| ドキュメント更新 | ✅ 確定 |
| コメント修正 | ✅ 確定 |
| コードリファクタリング（機能変更なし） | ✅ 確定 |

**特性（すべて ✅ 確定）**

- **総累積管理** — リビジョン番号はプロジェクト開始からの総変更回数を示す
- **リセット禁止** — ❌ いかなる理由があってもリビジョン番号をリセットしてはならない
- **連続性の保証** — プロジェクト全体の変更履歴について完全な追跡可能性を維持する

## バージョンアップの実例

### 例1: 初期開発からリリースまで

```
Ver.0.1-1  → 初期開発開始
Ver.0.1-2  → ドキュメント更新
Ver.0.2-3  → 機能追加
Ver.0.2-4  → バグ修正（軽微）
Ver.1.0-5  → 正式リリース
```

### 例2: マイナーアップデート

```
Ver.1.0-5  → 正式リリース
Ver.1.0-6  → バグ修正（軽微）
Ver.1.1-7  → 新機能追加
Ver.1.1-8  → ドキュメント更新
Ver.1.2-9  → 機能改良
```

### 例3: メジャーアップデート

```
Ver.1.5-15 → 現行バージョン
Ver.1.5-16 → 最終調整
Ver.2.0-17 → メジャーアップデート（破壊的変更）
Ver.2.0-18 → バグ修正
Ver.2.1-19 → 新機能追加
```

## まとめ

Adlaire Group のバージョニング方式は、以下の特徴を持つ。

| 特徴 | 内容 | ステータス |
|------|------|-----------| 
| セマンティックバージョニングベース | 変更の性質（メジャー、マイナー、リビジョン）を明確に表現する | ✅ 確定 |
| 総累積管理 | リビジョン番号はプロジェクト開始からの総変更回数を表す | ✅ 確定 |
| リセット厳禁 | いかなる場合もリビジョン番号をリセットしてはならない | ❌ 禁止 |
| 連続性の保証 | プロジェクト全体の変更履歴を永続的に追跡できる | ✅ 確定 |
| 明確な表記 | `Ver.{メジャー}.{マイナー}-{リビジョン}` 形式で表記する | ✅ 確定 |
