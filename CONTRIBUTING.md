## コントリビューションの流れ

### 1. Issue の作成

バグ報告・機能提案は、まず GitHub Issue を作成してください。

**バグ報告に含める情報:**
- 再現手順
- 期待される動作と実際の動作
- PHP バージョン・Web サーバー・OS
- 該当するエラーメッセージやログ

**機能提案に含める情報:**
- 提案の概要と目的
- 想定されるユースケース

### 2. プルリクエストの作成

1. リポジトリをフォーク
2. フィーチャーブランチを作成（`feature/説明` または `fix/説明`）
3. 変更を実装
4. プルリクエストを作成し、変更内容を説明

**プルリクエストのルール:**
- 1 つの PR につき 1 つの目的（バグ修正 or 機能追加）
- 既存のコードスタイルに従う
- 既存機能を壊さないことを確認する

### 3. レビューとマージ

- 権利者がレビューを行い、承認後にマージします
- 修正を求められた場合は、同じブランチに追加コミットしてください

## 開発環境のセットアップ

### 必要条件

- **PHP 7.4 以上**（推奨: PHP 8.1+）
- **Web サーバー**: Apache（mod_rewrite 有効）または PHP ビルトインサーバー

### ローカル起動

```bash
# PHP ビルトインサーバーで起動
php -S localhost:8000

# ブラウザで http://localhost:8000 にアクセス
```

### ディレクトリ構造

```
AdlairePlatform/
├── index.php              # エントリーポイント（ルーティング・コア関数）
├── engines/               # エンジンファイル（15 エンジン）
│   ├── AdminEngine.php    # 管理・認証・CSRF・編集・リビジョン
│   ├── ApiEngine.php      # REST API
│   ├── AppContext.php      # アプリケーション状態管理
│   ├── CollectionEngine.php
│   ├── GitEngine.php
│   ├── HookEngine.php
│   ├── JsEngine.php       # JS ファイル統合
│   ├── Logger.php          # 構造化ロギング
│   ├── MailerEngine.php    # メール送信
│   ├── MarkdownEngine.php
│   ├── MigrationEngine.php
│   ├── PluginEngine.php
│   ├── StaticEngine.php
│   ├── TemplateEngine.php  # テンプレートレンダリング
│   ├── ThemeEngine.php     # テーマ管理

├── themes/                # テーマ
└── docs/                  # ドキュメント
```

## コーディング規約

### PHP

- **外部依存なし** — Composer パッケージを追加しない
- **PHP 7.4 互換** — `match` 式や名前付き引数（PHP 8.0+）は使用しない
- **関数・メソッド名**: `camelCase`
- **クラス名**: `PascalCase`
- **エンジンパターン**: 新機能は `engines/` に独立したエンジンクラスとして追加
- **セキュリティ優先**: 入力バリデーション・出力エスケープ・CSRF 保護を徹底
- **グローバル変数の直接使用を避ける** — `AppContext` を経由する

### JavaScript

- **バニラ JS**（フレームワーク不使用）
- **ES2020+** 構文
- **モジュールシステム未使用** — `import`/`export` は使わない

### 全般

- インデントはスペース 4 つ
- 不要なコメントを追加しない — コードが自己説明的であること
- 既存のパターンに合わせる

## セキュリティに関するガイドライン

セキュリティ脆弱性を発見した場合は、**公開 Issue を作成しないでください**。
代わりに、権利者に直接連絡してください。

コントリビューション時に注意すべきセキュリティ項目:
- ユーザー入力は必ずバリデーション・サニタイズする
- HTML 出力には `h()` 関数でエスケープする
- 状態変更を伴うエンドポイントには CSRF トークン検証を含める
- ファイルパス操作にはパストラバーサル対策を施す
- メールヘッダーには `MailerEngine::sanitizeHeader()` を使用する

## ドキュメントの更新

コードの変更に伴い、以下のドキュメントも必要に応じて更新してください:

| ドキュメント | 更新が必要なケース |
|---|---|
| `RELEASE-NOTES.md` | すべての変更 |
| `README.md` | エンジン追加 |
| `docs/ARCHITECTURE.md` | アーキテクチャ・ファイル構成の変更 |
| `docs/SPEC.md` | API・テンプレート構文の変更 |
| `docs/features.md` | 機能追加・変更 |
