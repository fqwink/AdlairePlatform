## コントリビューションの流れ

### 1. Issue の作成

バグ報告・機能提案は、まず GitHub Issue を作成してください。

**バグ報告に含める情報:**
- 再現手順
- 期待される動作と実際の動作
- PHP バージョン・Web サーバー・OS
- 該当するエラーメッセージやログ

**機能提案に含める情報:**
- 提案の概要と目的
- 想定されるユースケース

### 2. プルリクエストの作成

1. リポジトリをフォーク
2. フィーチャーブランチを作成（`feature/説明` または `fix/説明`）
3. 変更を実装
4. プルリクエストを作成し、変更内容を説明

**プルリクエストのルール:**
- 1 つの PR につき 1 つの目的（バグ修正 or 機能追加）
- 既存のコードスタイルに従う
- 既存機能を壊さないことを確認する

### 3. レビューとマージ

- 権利者がレビューを行い、承認後にマージします
- 修正を求められた場合は、同じブランチに追加コミットしてください

## 開発環境のセットアップ

### 必要条件

- **PHP 8.3 以上**（必須）— PHP 8.2 以前は非対応
- **Web サーバー**: Apache（mod_rewrite 有効）または PHP ビルトインサーバー

### ローカル起動

```bash
# PHP ビルトインサーバーで起動
php -S localhost:8000

# ブラウザで http://localhost:8000 にアクセス
```

### ディレクトリ構造

詳細は [docs/ARCHITECTURE.md](docs/ARCHITECTURE.md) を参照してください。

## コーディング規約

### PHP

- **外部依存なし** — Composer パッケージを追加しない
- **PHP 8.3+ 対応** — PHP 8.3 以降の機能を使用可能
- **関数・メソッド名**: `camelCase`
- **クラス名**: `PascalCase`
- **Framework パターン**: 新機能は `Framework/` 内の該当モジュールに追加
- **セキュリティ優先**: 入力バリデーション・出力エスケープ・CSRF 保護を徹底
- **グローバル変数の直接使用を避ける** — `AppContext` を経由する

### JavaScript

- **バニラ JS**（フレームワーク不使用）
- **ES2020+** 構文
- **モジュールシステム未使用** — `import`/`export` は使わない

### 全般

- インデントはスペース 4 つ
- 不要なコメントを追加しない — コードが自己説明的であること
- 既存のパターンに合わせる

## セキュリティに関するガイドライン

セキュリティ脆弱性を発見した場合は、**公開 Issue を作成しないでください**。
代わりに、権利者に直接連絡してください。

コントリビューション時に注意すべきセキュリティ項目:
- ユーザー入力は必ずバリデーション・サニタイズする
- HTML 出力には `h()` 関数でエスケープする
- 状態変更を伴うエンドポイントには CSRF トークン検証を含める
- ファイルパス操作にはパストラバーサル対策を施す
- メールヘッダーには `MailerEngine::sanitizeHeader()` を使用する

### メタデータ記載ルール

全ドキュメントのファイル先頭に以下を必ず記載する。

| キー | 形式 | 例 |
|-----|------|----|
| ファイル名 | テキスト | `ARCHITECTURE.md` |
| バージョン | `Ver.{メジャー}.{マイナー}-{ビルド}` | `Ver.1.0-1` |
| 最終更新 | `YYYY-MM-DD` | `2026-03-10` |

- 日付フォーマットは `YYYY-MM-DD` に統一する

### 変更履歴記載ルール

全ドキュメントのファイル末尾に以下の変更履歴テーブルを必ず記載する。

```
## 変更履歴

| バージョン | 更新日 | 変更内容 |
|-----------|--------|----------|
| Ver.x.x-x | YYYY-MM-DD | 初版作成 |
```

- 更新のたびに行を追記する（既存行は削除しない）
- 新しい行を先頭に追加する（降順）
- `変更内容` は簡潔に1行で記載する

### バージョン番号の扱い

- ドキュメントのバージョンは [docs/VERSIONING.md](docs/VERSIONING.md) の累積バージョニング規則に従う
- ビルド番号はドキュメント単位で管理し、リセット禁止

### コード変更に伴うドキュメント更新

| ドキュメント | 更新が必要なケース |
|------------|-----------------|
| `RELEASE-NOTES.md` | すべての変更 |
| `README.md` | エンジン追加 |
